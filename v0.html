What is an Organisation?
  • An org is a named group of pre-configured deep links managed by admin
  • Example: "ACT Fibernet" org with links like "300Mbps Plans page", "App download"
  • Org has a password — regular users must enter it to access the org's links
  • This prevents any user from pulling private org deep link configs
What does an Org Link contain?
  • name — e.g. "Plans Page - 300mbps"
  • androidPath, androidScheme, androidPackage — intent URL components
  • androidPlaystore — Play Store fallback URL
  • androidWebUrl — web fallback for Android
  • androidIntentUrl — computed intent:// URL (stored)
  • iosAppStoreUrl, iosWebUrl — iOS config
How regular users access org links

Android redirect logic (in order)
  • 1. If androidUrl is set and is an intent:// URL → redirect to it (browser opens app or falls back to Play Store). Link data params are injected into the intent path.
  • 2. Else if androidWebUrl is set → redirect to it (with link data params appended)
  • 3. Fallback → redirect to longurl (the original web URL, with link data params)
iOS redirect logic (in order)
  • 1. If iosUrl is set → redirect to it (App Store URL or custom scheme)
  • 2. Else if iosWebUrl is set → redirect to it (with link data params)
  • 3. Fallback → redirect to longurl (with link data params)
Desktop redirect logic
  • Redirects to longurl (the original web URL)
  • If link data key-value pairs exist, they are appended as query params to the URL
  • Example: https://actfibernet.com/plans?offer=SAVE20&plan=300mbps
What gets logged per visit
  • timestamp — exact date/time
  • device — Android / iOS / Tablet / Desktop (from user-agent)
  • browser — Chrome / Firefox / Safari / Edge / Opera / Other
  • referrer — hostname of where the click came from (e.g. instagram.com, t.co, direct)

Why a separate store-redirect route?
  • Intent URLs can't log visits themselves — the browser handles them natively
  • By routing the fallback URL through the backend, the app can track how many times users were sent to the store
  • The storeRedirects array on each URL document stores these separately from normal visits
  • Visible in dashboard as "Store Redirects" column (A: count, i: count)
iOS store redirect
  • iOS App Store redirects use iosUrl (which is set to the App Store URL directly)
  • If it goes through /store-redirect/:urlId/ios, the same storeRedirects logging applies
